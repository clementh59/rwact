## What's Remaining for Production

### Smart Account Architecture (Critical)

**Current state:** One shared smart account (single owner private key in `.env`) used by all users. This works for demo but is not production-ready.

**Target architecture:**

1. User connects any ERC-4337 compatible smart account in the app (Kernel, Safe, Biconomy, etc.)
2. User approves a session key generated by the backend
3. Backend stores the session key and uses it to execute trades on behalf of the user
4. Each user has their own smart account with their own funds

**What this requires:**
- Frontend: wallet connection flow + session key approval UI
- Backend: accept external smart account addresses, generate session keys per user, validate session key permissions
- Remove the shared `OWNER_PRIVATE_KEY` from `.env`

### Other Production Gaps

- **Database**: Migrate from SQLite to PostgreSQL for multi-instance deployment
- **Rate limiting**: Debank API, Alpha Vantage, and SEC EDGAR calls are not rate-limited
- **Error recovery**: Background loops don't retry on transient failures
- **Workflow cleanup**: Disabled strategies don't stop their Otomato workflows (TODO in code)
- **Multi-stock SEC filings**: Currently hardcoded to Tesla's CIK only
- **Points API**: xStocks dashboard API returns "User not found" for most addresses (needs investigation)
- **Push token refresh**: If user's Expo push token changes, Otomato workflows still have the old one baked in
- **Earnings dedup**: In-memory dedup resets on server restart (should move to DB)
- **Disclosure dedup**: Same — in-memory baseline resets on restart
